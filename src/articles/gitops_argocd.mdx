# Understanding GitOps architecture with ArgoCD

Before following this article you need to have a good understanding of : 
- Git & Github
- Kubernetes
- CI/CD pipelines

## Traditional DevOps
Traditionally, the entire DevOps life cycle was based on push based mechanism means a developer commits code changes with a external web hook, either with jenkins or any other cicd pipeline to send code trigger then the cicd pipeline runs, then we manually write a script to update any configuration for deployment, either on kubernetes or any other deployment platform and this was the entire process.

## With GitOps
Now, if we fast forward and see in today's time, we have gitops, which basically means **Git + Operations** means here the git becomes the single source of truth for our deployment.

Here in gitops architecture, we don't have to juggle between multiple configuration tools, multiple cicd tools, or we don't even need to add a push based mechanism because gitops is pull based by default means whenever a new code change will be applied in your github repository then the new changes will be automatically be applied to your deployment.

## What we will build ?
To practically understand how gitops architecture actually works, we will use a tool called **ArgoCD**, which is a kubernetes native gitops tool. And here we have a two tier application, a frontend which is built on react, and a backend which is built on nodejs and express with a database mongodb. Now we will be deploying that backend on kubernetes with the help of argocd and github actions and will setup the entire GitOps pipeline.

![diagram](https://ik.imagekit.io/isudiptodas/portfolio-article-images/argocd-gitops-diagram.png)

1. Developer will push code changes in a repository
2. Github actions workflow will run and perform the specified jobs
3. Update kubernetes manifest file
4. ArgoCD will keep monitoring the manifest files so that the desired state and deployed state should be in sync

## Backend k8s manifests
Inside the `k8s` I have added two manifest files, one for deployment and another for service (loadbalancer)

`k8s/deployment.yaml`
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: form-builder-backend
  namespace: default
spec:
  replicas: 3
  selector:
    matchLabels:
      app: formbuilder
  template:
    metadata:
      labels:
        app: formbuilder
    spec:
      containers:
      - name: form-builder-backend
        image: isudiptodas/form-builder:1
        ports:
        - containerPort: 5000
        envFrom:
          - secretRef:
              name: formbuilder-secrets
        readinessProbe:
            httpGet:
              path: /health
              port: 5000
```
`k8s/service.yaml`
```yaml
apiVersion: v1
kind: Service
metadata:
  name: formbuilder-service
spec:
  type: LoadBalancer
  selector:
    app: formbuilder
  ports:
    - protocol: TCP
      port: 80
      targetPort: 5000
```


## Github actions configuration

```yaml
name: FormBuilder-Actions

on:
  push:
    branches:
      - main
    paths:
      - 'backend/**'

permissions:
  contents: write

jobs:
  build-test-push:
    runs-on: ubuntu-latest

    steps:
      # 1. Checkout
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      # 2. Setup Node
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22

      # 3. Install dependencies
      - name: Install dependencies
        working-directory: backend
        run: npm ci

      # 4. Run tests
      - name: Run tests
        working-directory: backend
        env:
            NODE_ENV: test
            MONGO_TEST_URI: ${{ secrets.MONGO_TEST_URI }}
            JWT_SECRET: ${{ secrets.JWT_SECRET }}
        run: npm test

      # 5. Docker Login
      - name: Login to DockerHub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      # 6. Build Docker Image
      - name: Build Docker Image
        run: |
          docker build \
            -t ${{ secrets.IMAGE_TAG }}:${{ github.run_number }} \
            ./backend

      # 7. Push Image
      - name: Push Docker Image
        run: |
          docker push ${{ secrets.IMAGE_TAG }}:${{ github.run_number }}

      # 8. Logout
      - name: Docker Logout
        run: docker logout

      # 9. Update K8s Manifest
      - name: Update deployment image
        run: |
          sed -i "s|image: .*|image: ${{ secrets.IMAGE_TAG }}:${{ github.run_number }}|g" backend/k8s/deployment.yaml

      # 10. Configure Git
      - name: Configure Git
        run: |
          git config user.name "isudiptodas"
          git config user.email "isudiptodas01@outlook.com"

      # 11. Commit changes
      - name: Commit manifest update
        run: |
          git add backend/k8s/deployment.yaml
          git commit -m "Updated k8s image tag [skip ci]" || echo "No changes to commit"

      # 12. Push changes
      - name: Push changes
        run: |
          git push origin main
```
Now here in the github actions configuration, you will see in many places I have added a dynamic variable, which I am using with the keyword **secrets**. Now you may wonder that what is a secret and what is even its value, and from where I am getting it. So just for the understanding in my script, I needed some values which are environment variables actually. And I need it because the script will test my application where I will need my mongodb URL, my jwt secret, and even while deploying my application on docker, I will need some credentials for that. So the secrets are referring to those actual environment values which I have stored in the github repository secrets.

## What about secrets ?

Now, if you have followed the entire article step by step, you may ask one thing that is while testing in github actions workflow, I have added the env in the repository secrets. But while deploying my application in kubernetes, where the secrets gone? Like will I not need to add secrets there or how the secrets will be managed in this gitops architecture if we need to add any secrets.

If this question came in your mind, then here is the perfect answer for that.

As we are implementing gitops architecture. And here git becomes the single source of growth for our application. So we can't really just upload our environment variables in github which will be very much risky. So we have two options, although we have a lot of options, but specifically industry standard, there are two options.

- First, you can encode your environment variables in basic base64 format and then create a resource of type **Secret** and you can deploy manually in the kubernetes cluster with the help of EC2.

- Or, you can use external plugins like : AWS secret manager or hashicorp vault and there you store the env variables and in the Secret manifest file you will just add the value paths instead of the base64 format and you can upload that file in github safely from where the argocd will apply the changes.
