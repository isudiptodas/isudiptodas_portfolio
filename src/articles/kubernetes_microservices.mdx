#  This is how microservices work on Kubernetes

The reason why I built this project is very simple because I wanted to learn how actually microservices work in production not just in theoretically but actually with real implementation. And as we all know, microservices means you are distributing your monolith or a single codebase into manageable individual services.

Before starting the overall discussion and implementation, I just want to clear out some things that 
- what have I built?
- why have I built this?
- how have I built this?
- what tech stack have I used?

## What have I built?
A calendar-based task management web app for managing tasks, automated reminders. Yeah, that's it, nothing special.

## Why have I built this?
Because I didn't find any more interesting topic to create on. And even my main aim was to implement microservice not to make revolutionary task management app.

## How have I built this?
At first, I have a **frontend**. Then instead of creating a single backend server I divided the backend based on each individual service, and now I have :

1. Authentication service (signup, login, user verification)
2. Task service (task CRUD operations)
3. Notification service (welcome email, automated reminders)
4. Password service (password recovery OTP, new password updation)

## What tech stack have I used?
 
1. **Frontend** : ReactJS + TypeScript
2. **Backend** : NodeJS + Express
3. **Database** : MongoDB
4. **Asynchronous tasks** : BullMQ + Redis
5. **Deployment** : AWS EKS (Kubernetes)

> The sole purpose of this article is to explain the implementation & working, so if you are hoping that every codefiles will be here then sorry for that, be ready for disappointment

## Architecture
| Development | Production |
|--|--|
| Frontend will use 4 separate API URLs to communicate with backend services | Frontend will communicate with only one single DNS with the help of ingress |

> **Ingress** :  An ingress in simple words is a route traffic controller, you define ingress with some routing configuration and it routes internet traffic to your service based on specified conditions / rules. 
> **NOTE** : You should use ingress only if you have multiple services running for a single application and you want path based routing otherwise normal load balancers will also work fine.

## Kubernetes example files

### 1. `k8s/auth-deployment.yaml`
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: auth-deployment
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels:
      app: auth
  template:
    metadata:
      labels:
        app: auth
    spec:
      containers:
        - name: auth-container
          image: <container-name>
          ports:
            - containerPort: 5000
          envFrom:
            - secretRef:
                name: taskcal-secrets
          readinessProbe:
            httpGet:
              path: /health
              port: 5000
```

This file is a Kubernetes `Deployment` example file for the `auth-service`.  
Its job is to tell Kubernetes how to run the auth service inside the cluster.

| Part | What it means |
|---|---|
| `apiVersion: apps/v1` | This tells Kubernetes which API version is being used for the Deployment resource. |
| `kind: Deployment` | This tells Kubernetes that we are creating a Deployment. A Deployment manages running copies of an application. |
| `metadata.name` | The name of this deployment is `auth-deployment`. |
| `metadata.namespace` | This deployment will be created inside the `default` namespace. |
| `spec.replicas: 2` | Kubernetes should run 2 copies of the auth service pod. If one crashes, Kubernetes can still keep another one running. |
| `selector.matchLabels.app: auth` | This tells the deployment which pods it should manage. It will manage pods with the label `app: auth`. |
| `template.metadata.labels.app: auth` | This label is attached to the pods created by this deployment. It must match the selector above. |
| `containers` | This section defines the container that will run inside the pod. |
| `name: auth-container` | The container name inside the pod. |
| `image: <container-name>` | The Docker image Kubernetes will pull and run for the auth service. In real deployment, this will be your pushed image name. |
| `ports.containerPort: 5000` | The auth service runs inside the container on port `5000`. |
| `envFrom.secretRef.name` | This loads environment variables from the Kubernetes secret named `taskcal-secrets`. |
| `readinessProbe` | Kubernetes uses this to check whether the auth service is ready to receive traffic. |
| `readinessProbe.httpGet.path: /health` | Kubernetes will call `/health` to check service readiness. |
| `readinessProbe.httpGet.port: 5000` | The health check request will be sent to port `5000`. |

In simple words, this file says:

> “Kubernetes, please run 2 copies of my auth service using this Docker image, expose port 5000 inside the pod, load environment variables from my secret file, and keep checking `/health` to know whether the service is ready.”


### 2. `k8s/auth-service.yaml`
```yaml
apiVersion: v1
kind: Service
metadata:
  name: auth-service
spec:
  type: ClusterIP
  selector:
    app: auth
  ports:
    - protocol: TCP
      port: 80
      targetPort: 5000
```

This file is a Kubernetes `Service` example file for the `auth-service`.

A deployment runs the actual pods, but pods are temporary. Their IP addresses can change whenever Kubernetes restarts or replaces them. So we create a `Service` to give those pods one stable internal address.

| Part | What it means |
|---|---|
| `apiVersion: v1` | This tells Kubernetes which API version is being used for the Service resource. |
| `kind: Service` | This tells Kubernetes that we are creating a Service. |
| `metadata.name` | The service name is `auth-service`. Other parts of the cluster can use this name to reach the auth service. |
| `spec.type: ClusterIP` | This service is only reachable inside the Kubernetes cluster. It is not directly exposed to the public internet. |
| `selector.app: auth` | This service will send traffic to pods that have the label `app: auth`. |
| `ports.protocol: TCP` | The service will use TCP protocol, which is the normal protocol for HTTP traffic. |
| `ports.port: 80` | This is the port exposed by the Kubernetes service internally. Other services or Ingress can call `auth-service` on port `80`. |
| `ports.targetPort: 5000` | This is the actual port where the auth container is running inside the pod. |

In simple words, this file says:

> “Kubernetes, create an internal stable network address called `auth-service`. Whenever someone sends traffic to this service on port `80`, forward it to the auth pods running on port `5000`.”

### 3. `k8s/secrets.yaml`
```yaml
apiVersion: v1
kind: Secret
metadata:
  name: taskcal-secrets
  namespace: default
type: Opaque
stringData:
  PORT:
  JWT_SECRET:
  MONGO_URI:
  REDIS_USERNAME:
  ....
  ....
  ....
  ....
```

This file is a Kubernetes `Secret` file.

A secret is used to store sensitive or environment-specific values like database URLs, JWT secret, Redis credentials, and email API keys. Instead of writing these values directly inside deployment files, we keep them in one secret and load them into services using `envFrom`.

| Part | What it means |
|---|---|
| `apiVersion: v1` | This tells Kubernetes which API version is being used for the Secret resource. |
| `kind: Secret` | This tells Kubernetes that we are creating a Secret. |
| `metadata.name` | The secret name is `taskcal-secrets`. Deployment files use this name to load environment variables. |
| `metadata.namespace` | This secret will be created inside the `default` namespace. |
| `type: Opaque` | This is the default general-purpose secret type in Kubernetes. |
| `stringData` | This lets us write secret values as normal strings. Kubernetes converts them internally into encoded secret data. |
| `PORT` | The port used by a service. |
| `JWT_SECRET` | Secret key used for signing and verifying JWT tokens. |
| `MONGO_URI` | MongoDB Atlas connection string. |
| `REDIS_USERNAME` | Redis Cloud username. |

In simple words, this file says:

> “Kubernetes, store all my environment variable names in one place, and let my backend services load them securely during runtime.”


### 4. `k8s/ingress.yaml`
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: taskcal-ingress
  namespace: default
spec:
  rules:
    - http:
        paths:
          - path: /api/auth
            pathType: Prefix
            backend:
              service:
                name: auth-service
                port:
                  number: 80
          - path: /api/task
            pathType: Prefix
            backend:
              service:
                name: task-service
                port:
                  number: 80
          - path: /api/password-recovery
            pathType: Prefix
            backend:
              service:
                name: password-service
                port:
                  number: 80
          - path: /api/notification
            pathType: Prefix
            backend:
              service:
                name: notification-service
                port:
                  number: 80
```

This file is a Kubernetes `Ingress` file for the whole application.

A service gives internal access inside the cluster, but Ingress gives controlled external access from outside the cluster. In this project, the frontend can call one single backend URL, and Ingress routes the request to the correct service based on the path.

| Part | What it means |
|---|---|
| `apiVersion: networking.k8s.io/v1` | This tells Kubernetes which API version is being used for the Ingress resource. |
| `kind: Ingress` | This tells Kubernetes that we are creating an Ingress. |
| `metadata.name` | The ingress name is `taskcal-ingress`. |
| `metadata.namespace` | This ingress will be created inside the `default` namespace. |
| `spec.rules` | This contains routing rules for incoming HTTP requests. |
| `http.paths` | This contains all path-based routing rules. | 
| `path: /api/auth` | Any request starting with `/api/auth` will go to the auth service. |
| `path: /api/task` | Any request starting with `/api/task` will go to the task service. |
| `path: /api/password-recovery` | Any request starting with `/api/password-recovery` will go to the password service. |
| `path: /api/notification` | Any request starting with `/api/notification` will go to the notification service. |
| `pathType: Prefix` | This means all URLs starting with that path will match. For example, `/api/auth/login` matches `/api/auth`. |
| `backend.service.name` | This is the Kubernetes service where the request should be forwarded. |
| `backend.service.port.number: 80` | This tells Ingress to send traffic to port `80` of that Kubernetes service. |

In simple words, this file says:

> “Kubernetes, expose one public entry point for my backend. If the request starts with `/api/auth`, send it to auth service. If it starts with `/api/task`, send it to task service. If it starts with `/api/password-recovery`, send it to password service. If it starts with `/api/notification`, send it to notification service.”

## Architecture in development
![diagram-before](https://ik.imagekit.io/isudiptodas/portfolio-article-images/taskcal-architecture-before.png)

-   The frontend directly calls every backend service separately, like auth, task, notification, and password recovery.
-   This works for development, but it is not ideal for production because the frontend needs to know multiple backend URLs.
-   Every service manages its own `.env`, so configuration becomes scattered and harder to control.
-   If any backend URL changes, the frontend configuration also needs to be updated.
-   This setup is fine for local testing, but it does not look like a clean production architecture.

## Architecture after deploying on kubernetes
![diagram-after](https://ik.imagekit.io/isudiptodas/portfolio-article-images/taskcal-architecture-after.png)

-   The user/frontend sends requests to one single entry point: the Kubernetes Ingress.
-   Ingress checks the API path, like `/api/auth` or `/api/task`, and forwards the request to the correct backend service.
-   The frontend no longer needs to know all individual backend service URLs.
-   All backend services can share common environment values through one Kubernetes Secret.
-   This setup is cleaner, easier to manage, and closer to how real production microservice systems are deployed.
